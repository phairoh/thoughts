Debugging / Hyprland / Wayland

Hyprland from LightDM: keyboard and mouse completely frozen

Vital signs are stable, the patient looks fine — but they're completely unresponsive. A diagnostic walkthrough for when your Wayland session starts but accepts no input.

EndeavourOS / Arch-based Hyprland 0.55+ LightDM + slick-greeter

Presentation

The patient — your Hyprland session — presents with a striking dissociation: motor function is intact but sensory input has completely failed. The wallpaper renders, Waybar is running, window decorations appear normal. Vitals look stable. And yet the keyboard and mouse are entirely unresponsive. No cursor movement. No reaction to keypresses. The session cannot be exited gracefully; a hard reset is the only option.

Critically, the system reports no obvious errors. Hyprland believes it started successfully. This is what makes the case deceptive — it looks like a healthy patient.

Differential diagnosis

The presenting symptoms — a process that is running but cannot receive input — suggest a problem with access to input devices at the system level. Two interventions appear indicated and are likely to be attempted first:

→

Intervention 1: Grant the user direct access to raw input devices via the input group — analogous to expanding a patient's care team permissions:

sudo usermod -aG input $USER

No effect. Symptoms unchanged.

→

Intervention 2: Enable seatd, an alternative seat management service, and add the user to the seat group — a more aggressive escalation of device access privileges:

sudo systemctl enable --now seatd
sudo usermod -aG seat $USER

Contraindicated. Condition deteriorates — the display stops loading entirely. Discontinue immediately and reverse:

sudo systemctl disable --now seatd

Both interventions target the wrong system. This is a classic diagnostic trap: treating the apparent site of dysfunction rather than the underlying cause. The input devices themselves are not the problem.

Localisation test

Before ordering further investigations, perform a localisation test to identify where in the chain the failure actually occurs. At the LightDM login screen, bypass the display manager entirely by switching to a TTY — think of this as removing one layer of the care pathway and assessing the patient directly:

Ctrl+Alt+F2        # switch to TTY
start-hyprland     # launch compositor directly

If the session works perfectly from the TTY — keyboard, mouse, display all functional — the underlying hardware and kernel-level permissions are healthy. The lesion is not in the patient. It is in the referral pathway: specifically, in how LightDM is handing the session off.

Aetiology

LightDM was designed in the X11 era and applies a legacy session wrapper script to every session it launches, regardless of type:

session-wrapper=/etc/lightdm/Xsession

This wrapper prepares an X11 environment and, crucially, sets XDG_SESSION_TYPE=x11 before passing control to the child process. Hyprland inherits this miscoded referral and behaves accordingly — it identifies itself to logind as an X11 session.

This is where input breaks down. Logind manages access to input devices and uses session type as a key part of that handoff. An X11 session type tells logind to expect an X server to manage input; logind will not hand device access to a Wayland compositor that has identified itself as X11. The compositor is running, but it has been administratively locked out of the input device queue — not because of missing permissions, but because it arrived under the wrong credentials.

Display output is unaffected because the GPU handoff pathway does not go through the same logind mechanism — which is why the visual presentation looks normal while input is completely absent.

Treatment

Install uwsm (Universal Wayland Session Manager):

sudo pacman -S uwsm

UWSM acts as a correctly-credentialed referring agent. It sets XDG_SESSION_TYPE=wayland, initialises the appropriate D-Bus and systemd user session environment, and ensures the compositor is presented to logind under the correct session type — so the input device handoff proceeds normally.

Once installed, a corrected session entry — Hyprland (uwsm-managed) — appears in the LightDM session menu. Select it. On next login, the patient will be fully responsive.

Why the entry already existed but didn't work

Hyprland ships a hyprland-uwsm.desktop session file alongside its standard hyprland.desktop. The uwsm-managed option was already visible in the LightDM session list — the referral pathway existed on paper — but the service it pointed to (uwsm) wasn't installed. A listed treatment that hasn't been stocked in the pharmacy. Installing the package activates it.

Affected populations

This presentation occurs in any configuration where a Wayland compositor is referred through LightDM without a Wayland-aware session wrapper. It is not specific to Hyprland, nor to slick-greeter — the Xsession wrapper is a LightDM behaviour that affects all compositors launched through it the same way.

Patients managed through GDM are not affected. GDM has native Wayland session support and handles the environment handoff correctly without intervention.

Prophylaxis

Include uwsm in the initial install to prevent the condition from presenting at all:

sudo pacman -S hyprland waybar wofi kitty dunst \
  xdg-desktop-portal-hyprland xdg-desktop-portal-gtk \
  polkit-gnome pipewire pipewire-alsa pipewire-pulse wireplumber \
  qt5-wayland qt6-wayland hyprpaper grim slurp \
  ttf-jetbrains-mono uwsm hyprlock hypridle

The deceptive feature of this case is that the patient looks well — the session loaded, the compositor is running, no error has been raised. The TTY localisation test is the critical step: it confirms that hardware and kernel permissions are intact and shifts the diagnosis upstream to the referral pathway. Once the fault is correctly localised to LightDM's Xsession wrapper, the treatment is straightforward. Install uwsm, select the uwsm-managed session, discharge.
