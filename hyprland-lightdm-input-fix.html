Hyprland from LightDM: keyboard and mouse completely frozen

Display renders, session loads, compositor is running — but nothing responds. Here's why, and how to fix it without touching seat permissions.

EndeavourOS / Arch-based Hyprland 0.55+ LightDM + slick-greeter

Symptoms

After installing Hyprland and selecting it from the LightDM session menu, the compositor loads successfully: the wallpaper appears, Waybar renders, window decorations look correct. But the keyboard and mouse are completely unresponsive. No cursor movement. No keypress acknowledgment. The only exit is a hard reset.

Notably, there are no obvious error messages — Hyprland appears to have started correctly, which makes this especially frustrating to diagnose.

Dead ends worth skipping

The symptoms point toward an input device permissions problem, so the natural first attempts are:

→

Adding the user to the input group:

sudo usermod -aG input $USER

No effect.

→

Enabling seatd and adding to the seat group:

sudo systemctl enable --now seatd
sudo usermod -aG seat $USER

Made things worse — the display stopped loading entirely while the mouse cursor briefly flashed. Revert before continuing:

sudo systemctl disable --now seatd

Both are wrong tracks. The problem isn't seat or input group membership.

The diagnostic that matters

Before spending more time on config, run this test: at the LightDM login screen, switch to a TTY with Ctrl+Alt+F2 and launch Hyprland directly:

start-hyprland

If keyboard and mouse work perfectly from the TTY — display, input, everything — then the system-level hardware and permissions are fine. The problem is specific to how LightDM is handing off the session. That's the key insight.

Root cause

LightDM applies a session wrapper script to all session types, configured via:

session-wrapper=/etc/lightdm/Xsession

This wrapper was written for X11 and sets up an X11 environment before launching the child process — including exporting XDG_SESSION_TYPE=x11. Hyprland inherits this environment variable and, seeing an X11 session type, cannot properly claim input devices from logind. Logind's input device handoff is session-type-aware; an X11 session type means it expects an X server to manage input, not a Wayland compositor.

The result is a compositor that renders correctly (display output doesn't go through logind in the same way) but has no access to keyboard or mouse events.

Fix

Install uwsm (Universal Wayland Session Manager):

sudo pacman -S uwsm

UWSM is designed specifically to handle the environment setup for Wayland compositors launched from display managers. It correctly sets XDG_SESSION_TYPE=wayland, initializes the appropriate D-Bus and systemd user session environment, and bypasses the Xsession wrapper.

Once installed, a new entry — Hyprland (uwsm-managed) — appears in the LightDM session selection menu alongside the standard Hyprland entry. Select that one. Keyboard and mouse will work on the next login.

Why the standard Hyprland entry still exists

Hyprland ships a hyprland-uwsm.desktop session file alongside its standard hyprland.desktop. The uwsm-managed entry was already present in the session list — it just didn't work until the uwsm package was actually installed. If you installed Hyprland via pacman -S hyprland and didn't explicitly install uwsm, the entry is there but silently broken.

Affected configurations

This issue affects any setup where a Wayland compositor is launched through LightDM without a Wayland-aware session wrapper. It's not specific to Hyprland — other compositors launched through LightDM the same way would exhibit the same problem. It's also not specific to slick-greeter; the wrapper script is a LightDM behavior, not a greeter behavior.

If you're on a display manager with native Wayland session support (GDM, for example), this doesn't apply — GDM handles Wayland session environment setup correctly by default.

Full install command for reference

The complete initial package install, including uwsm from the start:

sudo pacman -S hyprland waybar wofi kitty dunst \
  xdg-desktop-portal-hyprland xdg-desktop-portal-gtk \
  polkit-gnome pipewire pipewire-alsa pipewire-pulse wireplumber \
  qt5-wayland qt6-wayland hyprpaper grim slurp \
  ttf-jetbrains-mono uwsm hyprlock hypridle

Adding uwsm here from the start avoids the issue entirely.

The misleading part of this bug is that everything looks like it worked — the compositor loaded, the session is running. The TTY test is the fastest way to isolate whether you're dealing with a hardware/permissions issue or a session environment issue. If TTY works and LightDM doesn't, it's the wrapper. Install uwsm, select the uwsm-managed session, done.
